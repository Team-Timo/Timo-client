## 트리거

- "리팩터링 필요해? / 이 코드 개선해줘"
- 기능 추가 없이 코드 구조 개선이 목적인 경우

## 참조

- `docs/conventions.md` → 함수·파일 크기 기준
- `docs/architecture.md` → 컴포넌트 계층, 모듈 경계

## 워크플로우

### Phase 1 — 필요성 판단
아래 중 하나라도 해당하면 리팩터링 권고한다:
- 함수 50줄 초과
- 파일 800줄 초과
- 동일 로직이 3곳 이상 중복
- 컴포넌트 계층 위반
- 하나의 컴포넌트가 너무 많은 관심사 처리

### Phase 2 — 범위 정의
리팩터링 대상을 명확히 하고, 기능 변경 없이 구조만 개선함을 명시한다.

### Phase 3 — 우선순위 제시

| 우선순위 | 기준 |
|----------|------|
| 높음 | 버그 유발 가능성 있는 구조 문제 |
| 중간 | 가독성·유지보수성 저하 |
| 낮음 | 스타일·네이밍 정리 |

### Phase 4 — 실행 계획 제안
변경할 파일, 추출할 함수/컴포넌트, 커밋 단위를 제안하고 승인 후 실행한다.
