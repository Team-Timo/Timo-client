# timo-pr

## 트리거

- "PR 만들어줘", "PR 올려줘", "pull request"
- 브랜치 작업이 완료되고 `develop`으로 머지 준비가 된 시점

## 참조

- 템플릿: `.github/PULL_REQUEST_TEPLATE.md`
- 제목 컨벤션: `docs/conventions.md` → 이슈 & PR 제목 컨벤션

## 핵심 원칙

1. **전체 커밋 히스토리 기준**: `develop` 브랜치에서 갈라진 전체 변경을 분석한다. 최신 커밋 하나만 보지 않는다.
2. **제목은 `[TYPE] 한국어 요약` 형식**
3. **WHY 중심**: 무엇을 바꿨는지는 diff가 대신한다. 본문에는 왜 이 변경이 필요했는지를 적는다.
4. **base 브랜치 확인**: 기본값은 `develop`. 다르면 사용자에게 먼저 물어본다.

---

## Phase 1 — 컨텍스트 수집 및 계획

```bash
git log develop...HEAD --oneline      # 포함될 커밋 전체 확인
git diff develop...HEAD --stat        # 전체 변경 파일 파악
```

브랜치 이름에서 `{type}`과 `{scope}` 추출 (`feat/web/5-social-login` → `FEAT`, `web`, `#5`)  
브랜치 push 여부 확인 → 없으면 `git push -u origin <branch>` 수행.

---

## Phase 2 — PR 생성

```bash
gh pr create \
  --title "[FEAT] 소셜 로그인 기능 구현" \
  --base develop \
  --body "$(cat <<'EOF'
## ISSUE 🔗

close #5

<br><br>

## What is this PR? 🔍

...

<br><br>

## To Reviewers

...

## Screenshot 📷

<!-- 구현된 기능/디자인 gif -->

<br><br>

## Test Checklist ✔

- [ ] 기능 동작 확인
- [ ] 엣지 케이스 확인
EOF
)"
```

---

## PR 제목 형식

```
[TYPE] <전체 변경사항을 아우르는 한국어 요약>
```

TYPE은 작업 성격에 맞게 대문자로 자유롭게 지정한다.  
(`[FEAT]`, `[FIX]`, `[REFACTOR]`, `[CI]`, `[INIT]` 등)

---

## 본문 작성 규칙

`.github/PULL_REQUEST_TEPLATE.md`의 모든 섹션을 빠짐없이 채운다.

| 섹션                  | 채우는 방법                                         |
| --------------------- | --------------------------------------------------- |
| `ISSUE 🔗`            | 브랜치명에서 추출한 이슈 번호 (`close #5`)          |
| `What is this PR? 🔍` | 아래 작성 방식 참고                                 |
| `To Reviewers`        | 본문 안에 자연스럽게 녹여 씀 (별도 항목 구조 금지)  |
| `Screenshot 📷`       | UI 변경이 있으면 표 형태로 구성, 없으면 주석만 유지 |
| `Test Checklist ✔`    | 검증이 필요한 항목을 구체적으로 작성                |

### What is this PR? 작성 방식

**구조 항목 금지.** 자연스러운 서술체로 작성한다.

1. **첫 문단**: 이번 PR에서 무엇을 했는지 전체를 한두 문장으로 요약한다.
2. **섹션 분리**: 변경된 도메인·컴포넌트·기능 단위로 `## 섹션명` 헤더를 나눠 서술한다.
3. **이유 포함**: 각 섹션 안에서 왜 그렇게 했는지 이유를 자연스럽게 녹인다.
4. **리뷰 포인트**: 리뷰어가 봐야 할 부분이나 판단이 필요한 부분은 해당 섹션 안에 구어체로 포함한다.

```markdown
## What is this PR? 🔍

로그인 페이지의 반응형 레이아웃을 수정하고, 토큰 갱신 로직에서 발생하던 경쟁 조건을 해결했습니다.

## 로그인 페이지

| 컴포넌트        | 화면       |
| --------------- | ---------- |
| `LoginForm.tsx` | (스크린샷) |

모바일에서 폼 너비가 뷰포트를 넘어가는 문제가 있어 max-w 기준을 lg로 통일했습니다.

## 토큰 갱신

리프레시 요청이 동시에 여러 번 발생하던 문제를 큐 방식으로 해결했습니다.
interceptor 내부 로직이 조금 복잡해졌는데 한번 봐주세요.
```

### 작성 스타일

- **서술형**: "~했습니다", "~되었습니다" 체 사용. 명사형·축약형 금지.
  - 나쁜 예: "토큰 갱신 로직 수정" → 좋은 예: "토큰 갱신 로직을 수정했습니다"
- **불필요한 수식어 제거**: "~를 진행했습니다", "~에 대해서" 금지.
- **스크린샷 표**: UI 변경이 여러 컴포넌트에 걸칠 경우 `| 컴포넌트 | 화면 |` 표로 정리한다.

### 사전 검증 확인

PR 본문 완성 전, 아래가 통과됐는지 확인한다. 미통과 시 먼저 수정을 권고한다:

- `pnpm check-types`
- `pnpm lint`

---

## 금지

- PR을 merge하지 않는다.
- force-push로 base 브랜치를 덮어쓰지 않는다.
- `[TYPE]` 형식을 반드시 대괄호로 감싸 사용한다.
- reviewer, milestone은 사용자가 명시한 경우에만 추가한다.
